Unit 7: Security [3 Hrs.]

7.1 Needs of Security

Database security is essential to protect data from unauthorized access, misuse, corruption, or theft while ensuring availability and integrity for legitimate users.

Why Database Security is Critical

  • Databases store highly sensitive information: personal data (name, address, phone, email), financial records (bank accounts, credit cards), medical history, intellectual property, government records, passwords (hashed), etc.
  • Legal & regulatory compliance: GDPR, HIPAA, PCI-DSS, Nepal's Electronic Transactions Act, Data Protection Bill (ongoing), ISO 27001
  • Financial & reputational damage: Data breaches lead to lawsuits, fines, loss of trust (e.g., Equifax 2017, Yahoo 2013–2014, Bangladesh Bank 2016)
  • Insider threats: Employees, contractors, or partners with legitimate access can misuse data
  • Cyber threats: SQL injection, ransomware, credential stuffing, DDoS, phishing
  • Business continuity: Prevent downtime due to malicious deletion or corruption
Core Security Goals (CIA Triad + more):
• Confidentiality – prevent unauthorized disclosure
• Integrity – prevent unauthorized modification
• Availability – ensure system is usable when needed
• Authenticity – verify identity of users/systems
• Non-repudiation – prove actions cannot be denied

7.3 Access Control

Access control determines **who** can perform **what** operations on **which** objects.

Access Control Models

ModelDescriptionUsed InExample
Discretionary Access Control (DAC)Owner decides permissionsMost relational DBMS (GRANT/REVOKE)Alice grants SELECT on her table to Bob
Mandatory Access Control (MAC)System enforces based on labels & clearanceHigh-security (military, government)Top Secret data only accessible to cleared users
Role-Based Access Control (RBAC)Permissions assigned to roles, users get rolesEnterprise systems (most common today)Role "HR Manager" has SELECT on EmployeeSalary
Attribute-Based Access Control (ABAC)Dynamic rules based on attributesModern cloud, fine-grained accessAllow access if user.department = 'HR' AND time between 9 AM–5 PM

Typical DBMS Access Control Layers:

  • Authentication (who are you?) – username/password, certificates, SSO, Kerberos
  • Authorization (what can you do?) – privileges on objects
  • Auditing (what did you do?) – log actions

7.4 Authorization

Authorization in relational DBMS is managed using GRANT and REVOKE commands with privileges.

Common Privileges

  • SELECT, INSERT, UPDATE, DELETE – on tables/views
  • EXECUTE – on stored procedures/functions
  • REFERENCES – create foreign keys
  • ALTER, DROP, INDEX, TRIGGER – structural changes
  • ALL PRIVILEGES – everything
  • CREATE SESSION / CONNECT – login
  • CREATE TABLE, CREATE VIEW, CREATE PROCEDURE – schema objects
-- Authorization Examples (PostgreSQL / MySQL / Oracle style)

-- Grant specific privileges
GRANT SELECT, INSERT ON Student TO ram;
GRANT SELECT ON Student TO PUBLIC;           -- everyone can read
GRANT ALL PRIVILEGES ON Department TO sita WITH GRANT OPTION;  -- can grant further

-- Role-based
CREATE ROLE hr_manager;
GRANT SELECT ON Employee, Salary TO hr_manager;
GRANT hr_manager TO ram, sita;

-- Revoke
REVOKE INSERT ON Student FROM ram;
REVOKE ALL PRIVILEGES ON Department FROM sita CASCADE;  -- revoke from others too
WITH GRANT OPTION / WITH ADMIN OPTION: Allows receiver to pass privilege to others.
Best Practice: Follow principle of least privilege – grant only necessary access.

7.5 Security and Views

Views are powerful tools for implementing security through data abstraction and access restriction.

Security Benefits of Views

  • Hide sensitive columns (projection)
  • Restrict rows (selection / WHERE clause)
  • Present simplified or joined data
  • Enforce row-level security without complex privileges
  • Allow different views for different roles
-- Security via Views

-- Hide salary from normal employees
CREATE VIEW EmployeePublic AS
SELECT emp_id, name, dept, hire_date
FROM Employee;

-- Row-level: only show own department
CREATE VIEW DeptEmployee AS
SELECT * FROM Employee
WHERE dept = CURRENT_USER_DEPT();   -- or session variable

-- Grant access only to view
GRANT SELECT ON EmployeePublic TO PUBLIC;
GRANT SELECT ON DeptEmployee TO dept_manager_role;
Updatable Views: Some views allow UPDATE/INSERT/DELETE if they meet criteria (single table, no aggregate, no join in some DBMS).

7.6 Encryption and Decryption

Encryption protects data confidentiality at rest (stored) and in transit (network).

Types of Encryption in DBMS

TypeScopeWho Manages Keys?Example Tools / Features
Transparent Data Encryption (TDE)Entire database / tablespaceDBMS / OSOracle TDE, SQL Server TDE, PostgreSQL pgcrypto + filesystem encryption
Column-level EncryptionSpecific sensitive columnsApplication or DBMSENCRYPT(), AES_ENCRYPT() in MySQL
Application-level EncryptionBefore sending to DBApplicationClient-side encryption (always encrypted in SQL Server)
Data in TransitNetwork trafficSSL/TLSSSL/TLS certificates, require_secure_transport
-- MySQL / MariaDB column encryption example
CREATE TABLE Patient (
    patient_id INT PRIMARY KEY,
    name VARCHAR(100),
    disease_enc VARBINARY(255)   -- encrypted
);

INSERT INTO Patient (patient_id, name, disease_enc)
VALUES (1, 'Ram', AES_ENCRYPT('Diabetes', 'secret_key_32_bytes_long'));

SELECT name, AES_DECRYPT(disease_enc, 'secret_key_32_bytes_long') AS disease
FROM Patient;

Key Management Challenges

  • Where to store encryption keys? (HSM, key vault, environment variables)
  • Key rotation
  • Performance overhead (encryption/decryption cost)
  • Search & indexing on encrypted data (difficult → deterministic encryption or searchable encryption)
Important: Encrypting data does NOT protect against SQL injection or privilege abuse — combine with input validation, least privilege, and auditing.
