Unit 10: Object-oriented Database [2 Hrs.]

10.1 Introduction of OODMS

An **Object-Oriented Database Management System (OODMS / OODBMS)** is a database management system that directly supports object-oriented concepts such as classes, objects, inheritance, encapsulation, polymorphism, and complex object structures — without the need to map them to relational tables.

Historical Context:

  • 1980s–1990s: Rise of object-oriented programming (Smalltalk, C++, Java) → mismatch between OOP and RDBMS (impedance mismatch)
  • 1990s: Development of OODBMS standards (ODMG – Object Database Management Group, 1991–2001)
  • Commercial OODBMS: ObjectStore, Versant, GemStone/S, db4o, Cache (InterSystems), Objectivity/DB
  • Modern era: Object-relational DBMS (PostgreSQL, Oracle) + NoSQL document databases (MongoDB, CouchDB) adopted many OO ideas

Core Idea: Store objects directly as they are in memory — no need for object-relational mapping (ORM) layers like Hibernate or Entity Framework.

Key Difference from RDBMS:
RDBMS → normalized tables, joins, flat rows
OODBMS → persistent objects, direct navigation via pointers/references, complex types

10.2 Use, Benefit, Role of OODMS

Typical Use Cases / Application Areas

  • CAD/CAM systems (complex engineering designs, assemblies, hierarchies)
  • Geographic Information Systems (GIS) – spatial data, maps, topologies
  • Multimedia & Content Management – images, videos, documents with metadata
  • Telecommunications – call detail records, network topologies
  • Scientific & Engineering simulations – molecular structures, astronomical data
  • Real-time systems – financial trading, sensor data
  • Embedded systems & IoT – persistent object graphs

Benefits of OODBMS

BenefitDescriptionCompared to RDBMS
No Impedance MismatchObjects stored & retrieved as-isNo ORM layer → faster development, less code
Rich Data TypesSupports complex objects, collections, inheritanceRDBMS needs normalization + joins
Direct Pointer NavigationFast traversal of object graphsRDBMS requires joins (slower for deep hierarchies)
Versioning & Long TransactionsBuilt-in support for check-in/check-outHarder in RDBMS
Performance for Complex QueriesGood for navigational accessRDBMS excels at set-based queries
Encapsulation & ReuseMethods stored with dataRDBMS separates data & logic (stored procedures)

Role / Limitations in Modern Context

  • Role today: Niche use in specialized domains (CAD, scientific data, real-time systems)
  • Many OO features now available in Object-Relational DBMS (PostgreSQL JSONB, Oracle object types, SQL:1999 standard)
  • Document databases (MongoDB, CouchDB) and graph databases (Neo4j) inherited many OODBMS ideas
  • Decline reasons: Lack of universal standard (ODMG failed), SQL dominance, maturity of RDBMS + ORM tools
Not dead, but niche: Pure OODBMS are rare today — most applications use RDBMS + ORM or NoSQL document/graph stores.

10.3 Object-oriented Data Model

The object-oriented data model integrates object-oriented programming concepts with database persistence.

Core Components of OO Data Model

  • Object: Basic unit — has unique identity (OID), state (attributes/values), behavior (methods)
  • Class: Template / blueprint for objects (defines attributes & methods)
  • Inheritance / Generalization: Class hierarchy (is-a relationship)
    • Single inheritance, multiple inheritance (controversial in databases)
  • Encapsulation: Data & methods bundled together; private/public access
  • Polymorphism: Same method name behaves differently based on object type
  • Complex Objects: Objects can contain other objects (composition/aggregation)
  • Object Identity (OID): System-generated, immutable identifier (not primary key like in RDBMS)
  • Associations / Relationships: References / pointers between objects
  • Collections: Sets, lists, bags, arrays as attribute types

Comparison: OO Data Model vs Relational Data Model

FeatureObject-oriented Data ModelRelational Data Model
Data StructureComplex objects, hierarchiesFlat tables, rows, columns
IdentitySystem OIDUser-defined primary key
RelationshipsDirect references/pointersForeign keys + joins
BehaviorMethods stored with classStored procedures / external code
Query LanguageOQL (Object Query Language), Java-like queriesSQL (declarative)
NavigationPointer-based (fast for graphs)Join-based (set-oriented)
StandardizationWeak (ODMG 3.0 last major attempt)Strong (ANSI/ISO SQL)

Object Query Language (OQL) – Example (ODMG style)

// Simple OQL queries

SELECT s.name, s.address.street
FROM Student s
WHERE s.gpa > 3.5;

SELECT count(*)
FROM Department d, d.courses c
WHERE d.name = "Computer Science";

-- Path expression / navigation
SELECT c.instructor.name
FROM Course c
WHERE c.code = "CS101";
Modern equivalents:
• PostgreSQL → JSONB + composite types + inheritance
• MongoDB → embedded documents + references
• Graph databases (Neo4j) → nodes & relationships
These borrow heavily from OO concepts without being pure OODBMS.
