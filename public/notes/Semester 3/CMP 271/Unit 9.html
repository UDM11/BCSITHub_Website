9.2 Recovery and Atomicity

Recovery ensures that after any failure, the database returns to a consistent state while preserving ACID properties — especially **Atomicity** and **Durability**.

Recovery Goals

  • Atomicity: Either all operations of a transaction are reflected or none
  • Durability: Committed transactions survive crashes
  • Undo uncommitted transactions
  • Redo committed transactions that may not have reached disk

Steal vs Force Policy

PolicyDescriptionAdvantageDisadvantageUsed In
No-StealUncommitted changes never written to diskNo undo needed after crashHigh memory pressureRare
StealUncommitted changes can be written to diskBetter buffer utilizationUndo needed on recoveryMost DBMS
No-ForceCommitted changes need not be forced to disk immediatelyFaster commitRedo needed on recoveryMost DBMS
ForceCommitted changes forced to disk at commitNo redo neededSlow commitRare (high durability need)

Most common combination: Steal + No-Force → requires both undo and redo → achieved via **Write-Ahead Logging (WAL)**.

9.4 Log-based Recovery

Log-based recovery (most widely used) relies on a sequential log file that records every database change before it is applied to data pages.

Write-Ahead Logging (WAL) Rule

  • Log record must be flushed to stable storage before corresponding data page is written
  • Log record for commit must be flushed before transaction is considered committed

Recovery Phases (ARIES Algorithm – Industry Standard)

  1. Analysis Phase: Scan log from last checkpoint forward
    • Build Dirty Page Table
    • Build Transaction Table (active transactions)
  2. Redo Phase: Redo all committed changes from earliest dirty page LSN
    • Redo even if page already has change (idempotent)
  3. Undo Phase: Undo uncommitted transactions (backward from end)
    • CLR (Compensation Log Record) written for each undo
ARIES advantages: Fast recovery (only redo from min RecLSN), supports fuzzy checkpointing, handles partial transaction rollback.

9.5 Shadow Paging

Shadow paging is an alternative to log-based recovery that avoids logging data changes by maintaining two copies of the database page table (current & shadow).

How Shadow Paging Works

  1. Two page tables: current & shadow (initially identical)
  2. When a transaction modifies a page:
    • Copy modified page to new disk location
    • Update shadow page table to point to new location
  3. At commit:
    • Atomically swap current ↔ shadow page table pointer (single disk write)
    • Discard old pages (garbage collection later)
  4. At crash: discard shadow table → old current table remains consistent

Advantages & Disadvantages

AspectAdvantageDisadvantage
RecoveryVery fast (no redo/undo)—
AtomicitySingle page-table swap—
OverheadNo logging of data changesCopy-on-write → high I/O for updates
ConcurrencyDifficult (needs multi-version or locking)Limited support in practice
UseEarly systems, some file systems (ZFS, Btrfs)Rare in modern relational DBMS
Shadow paging is rarely used in relational DBMS today — log-based (ARIES) is dominant due to better concurrency, lower update cost, and support for partial rollbacks.

9.6 Local Recovery Manager

The **Local Recovery Manager (LRM)** is the component within a DBMS responsible for ensuring atomicity and durability for transactions on a single node (in centralized or distributed systems).

Responsibilities of Local Recovery Manager

  • Write-ahead logging of all updates
  • Flush log records at commit
  • Perform checkpointing (fuzzy or crisp)
  • Execute recovery after crash:
    • Analysis → Redo → Undo
  • Handle transaction rollback (undo changes)
  • Manage log buffer & log file
  • Coordinate with buffer manager (force/steal policy)

In distributed databases: Each site has its own LRM; global recovery uses 2PC (Two-Phase Commit) coordinator.

9.7 Backup System

Backup is the last line of defense against media failure, logical corruption, accidental deletion, or ransomware.

Types of Backup

TypeDescriptionBackup SizeRecovery TimeFrequency
Full BackupComplete copy of entire databaseLargeFastWeekly / Monthly
Differential BackupChanges since last full backupMediumMediumDaily
Incremental BackupChanges since last backup (full or incremental)SmallestSlowest (apply all incrementals)Multiple times/day
Log BackupTransaction log since last log backupSmallPoint-in-time recoveryEvery 5–60 min

Recovery Strategies Using Backup

  1. Restore last full backup
  2. Apply differential (if any) or all incrementals in order
  3. Apply log backups (roll forward) to desired point-in-time
  4. Recovery to consistent state using redo/undo from logs

Modern Backup Techniques

  • Snapshot backups (storage-level – fast, low overhead)
  • Continuous archiving (WAL shipping to standby server)
  • Hot / Online backup (backup while database running)
  • Cloud backups (AWS RDS snapshots, Azure Backup, Google Cloud)
Best Practice (3-2-1 Rule):
• 3 copies of data
• 2 different media types
• 1 copy offsite / off-cloud
