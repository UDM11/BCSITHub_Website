Unit 6: File Organization and Indexing [3 Hrs.]

6.1 Disks and Storage

Modern DBMS performance is heavily dependent on how data is physically stored and accessed on secondary storage (mainly disks), because disk I/O is orders of magnitude slower than main memory access.

Disk Structure (Magnetic Hard Disk Drive – HDD)

  • Platters: Circular disks coated with magnetic material
  • Tracks: Concentric circles on a platter
  • Sectors: Smallest addressable unit (typically 512 bytes or 4 KB)
  • Cylinders: Same track across all platters
  • Read/Write Heads: One per platter surface
  • Access Time = Seek time + Rotational latency + Transfer time

Typical Disk Parameters (2020s HDD)

ParameterTypical ValueNotes
Sector size512 B – 4 KBAdvanced Format = 4 KB
Seek time4–10 ms (average)Time to move head to track
Rotational latency2–5 ms (average)½ rotation at 7200–15,000 RPM
Transfer rate100–250 MB/sSequential read/write
Block size (DBMS)4 KB – 16 KBCommon page size

SSD (Solid State Drive) vs HDD

  • No moving parts → seek time ≈ 0.05–0.1 ms
  • Random I/O much faster
  • But: limited write endurance, higher cost per GB
  • Modern DBMS often use SSDs or hybrid storage
Key DBMS Insight: Disk I/O is the bottleneck in most database operations → goal is to minimize random I/O and maximize sequential access.

6.2 Organization of Records into Blocks

Data is stored on disk in fixed-size **blocks** (or pages) — the unit of I/O between disk and memory (typically 4 KB–16 KB in DBMS).

Types of Record Organization within a Block

TypeDescriptionAdvantagesDisadvantagesUse Case
Fixed-length recordsAll records same sizeEasy addressing, fast accessWaste space if variable dataSimple tables (integer keys)
Variable-length recordsDifferent sizesEfficient storageComplex management, fragmentationVARCHAR, TEXT columns
Packed recordsNo gaps between recordsMax storage efficiencyInsertion/deletion slowRead-heavy, static data
Unspanned recordsRecord cannot cross block boundarySimpleInternal fragmentationMost DBMS default
Spanned recordsRecord can span blocksNo fragmentation wasteComplex retrievalLarge objects (BLOBs)

Block Header Information (typical)

  • Block ID / Page ID
  • Number of records
  • Free space pointer / offset list
  • Checksum / LSN (Log Sequence Number)
  • Next / previous block pointer (for sequential files)
Slotted Page Structure: Most DBMS use slotted pages: header + slot array (pointers to records) + records from end of page → allows variable-length records with easy deletion (mark slot free).

6.3 File Organizations – Sequential and Indexed Sequential File Organizations

Heap File / Unordered File

  • Records inserted at end or in free space
  • No order → full table scan for search
  • Fast insert, slow search

Sequential File Organization

  • Records stored in order of search key (sorted)
  • Search: binary search possible if blocks are contiguous
  • Insertion: expensive (requires shifting or overflow area)
  • Common for reports / batch processing

Indexed Sequential Access Method (ISAM) / Indexed Sequential File

  • Records sorted on primary key
  • Primary index on blocks (sparse index)
  • Overflow area for inserts (chain of overflow blocks)
  • Degrades over time → requires periodic reorganization
  • Precursor to B+ tree indexes

// Conceptual ISAM structure
Level 0 (cylinder index) → points to track indexes
Level 1 (track index)   → points to block indexes
Level 2 (block index)   → points to data blocks + overflow chain

6.4 B+ Tree Index

B+ tree is the most widely used index structure in modern DBMS (InnoDB, PostgreSQL, Oracle, SQL Server, etc.).

Properties of B+ Tree

  • Balanced tree → all leaf nodes at same level
  • Internal nodes contain only keys & pointers (no data)
  • Leaf nodes contain keys + data pointers (or actual data in clustered index)
  • Nodes have fan-out (order) m: between ⌈m/2⌉ – m children (except root)
  • Leaf nodes linked sequentially → efficient range scan

Advantages over B-tree

  • Better range queries (sequential leaf links)
  • More keys per node (no data in internal nodes)
  • Stable performance (balanced height)

Typical Parameters

  • Node size = disk block (4–16 KB)
  • Fan-out ≈ 100–500 (depending on key size)
  • Height ≈ 3–4 for billions of records

// Example B+ tree (order = 3, i.e. 2–3 keys per node)
Root:          [50]
             /     \
Internal:   [20,30]   [60,80]
           /  |  \    /  |  \
Leaves: [10,15] [20,25,30] [40]   [50,55] [60,70] [80,90,100]
        ↑ linked sequentially →
Clustered vs Non-clustered:
Clustered (primary index): leaf nodes contain full rows (InnoDB default)
Non-clustered (secondary): leaf nodes contain primary key → extra lookup

6.5 Hash Index

Hash indexes map search key values to bucket addresses using a hash function.

Static Hashing

  • Fixed number of buckets
  • Hash(key) mod M → bucket number
  • Collisions handled by chaining or open addressing
  • Problem: under-utilization or overflow as data grows

Dynamic Hashing / Extendible Hashing

  • Directory of pointers to buckets
  • Global depth & local depth
  • Splits buckets & doubles directory when overflow
  • No full table reorganization
  • Used in PostgreSQL (hash index), SQL Server (memory-optimized tables)

Important: Most production DBMS prefer B+ trees over hash indexes for disk-based storage because range queries, sorting, and joins are common operations.
