Unit 3: Normalization [7 Hrs.]

3.1 Importance of Normalization

Normalization is a systematic process of organizing data in a database to reduce redundancy and prevent undesirable anomalies during data operations (INSERT, UPDATE, DELETE).

Problems in Unnormalized / Poorly Designed Tables (Insertion, Update, Deletion Anomalies)

Anomaly TypeDescriptionExample (Bad Design)
Insertion AnomalyCannot insert data without unrelated infoCannot add new faculty without assigning a course if faculty-course is in same table
Update AnomalyUpdating one fact requires multiple row changesChange faculty phone → update all rows where faculty appears
Deletion AnomalyDeleting one fact removes unrelated factsDelete last student of a course → lose course & instructor info

Benefits of Normalization

  • Eliminates / minimizes data redundancy
  • Reduces anomalies → improves data integrity
  • Makes database easier to maintain and extend
  • Improves consistency of data
  • Optimizes storage (in most cases)
  • Facilitates query writing (especially joins)
Trade-off: Normalization → more tables → more joins → can reduce read performance. Solution: Controlled denormalization for read-heavy systems (data warehouses, reporting tables).

3.2 Functional Dependencies – Definition, Trivial & Non-trivial FD, Closure of FD Set, Closure of Attributes

Functional Dependency (FD) – Definition

A functional dependency X → Y holds on relation R if for every valid instance of R, whenever two tuples agree on all attributes in X, they must also agree on all attributes in Y.

In other words: X uniquely determines Y.

Types of Functional Dependencies

  • Trivial FD: Y is a subset of X (always true)
    Example: {student_id, name} → name
  • Non-trivial FD: Y is not a subset of X
  • Completely non-trivial: X ∩ Y = ∅

Closure of a Set of Functional Dependencies (F⁺)

The closure F⁺ of a set F of FDs is the set of all FDs that can be logically inferred from F using Armstrong’s axioms.

Armstrong’s Axioms (Inference Rules)

  1. Reflexivity: If Y ⊆ X then X → Y (trivial)
  2. Augmentation: If X → Y then XZ → YZ
  3. Transitivity: If X → Y and Y → Z then X → Z
  4. Union: If X → Y and X → Z then X → YZ
  5. Decomposition: If X → YZ then X → Y and X → Z
  6. Pseudotransitivity: If X → Y and WY → Z then WX → Z

Attribute Closure (X⁺)

The closure of attribute set X (X⁺) is the set of all attributes that are functionally determined by X (directly or indirectly).


// Algorithm to compute closure of attributes X⁺ given FDs F
Input: Set of attributes X, Set of FDs F
Output: X⁺

1. result := X
2. repeat
     for each FD U → V in F do
       if U ⊆ result then
         result := result ∪ V
   until result does not change
3. return result

Example:
Relation R(A,B,C,D,E)
FDs: A→B, B→C, AC→D, E→A

Closure of {A,C}:
Step 1: {A,C}
Step 2: A→B → add B → {A,B,C}
        AC→D → add D → {A,B,C,D}
Step 3: B→C (already have)
Final: {A,B,C,D}
Use of closure: Check if X is superkey (X⁺ = all attributes), find candidate keys, test normalization.

3.4 Normal Forms (1NF, 2NF, 3NF, BCNF)

1NF – First Normal Form

  • All attributes are atomic (indivisible)
  • No repeating groups / multi-valued attributes in columns
  • Each column contains values of single domain
  • Usually includes: table has primary key

2NF – Second Normal Form

  • Table is in 1NF
  • No partial dependency: non-prime attribute is not dependent on proper subset of candidate key

3NF – Third Normal Form

  • Table is in 2NF
  • No transitive dependency: non-prime attribute does not depend on another non-prime attribute
  • Every non-key attribute is directly dependent on candidate key

BCNF – Boyce-Codd Normal Form (Stronger 3NF)

  • Table is in 3NF
  • For every non-trivial FD X → Y, X must be a superkey
  • Eliminates anomalies caused by overlapping candidate keys

Step-by-step Normalization Example


// Unnormalized → 1NF → 2NF → 3NF → BCNF

Unnormalized:
OrderID | CustomerName | CustomerCity | Product | Quantity | UnitPrice | SupplierName | SupplierCity

1NF (remove repeating groups, atomic values):
OrderID | CustomerName | CustomerCity | Product | Quantity | UnitPrice | SupplierName | SupplierCity

Candidate keys: {OrderID, Product}

2NF violation: Partial dependency CustomerName → CustomerCity (depends only on OrderID)
→ Split:

Orders: OrderID PK, CustomerName, CustomerCity
OrderDetails: OrderID FK, Product PK(composite), Quantity, UnitPrice, SupplierName, SupplierCity

3NF violation: SupplierName → SupplierCity (transitive)
→ Split further:

Suppliers: SupplierName PK, SupplierCity
OrderDetails: OrderID FK, Product PK, Quantity, UnitPrice, SupplierName FK

BCNF check: All determinants are keys → already in BCNF
Practical Tip: Most production databases aim for 3NF (balance between integrity & performance). BCNF is ideal when lossless join & dependency preservation are both required.
