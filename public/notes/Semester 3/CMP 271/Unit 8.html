8.1 Introduction to Transaction

A **transaction** is a logical unit of work that consists of one or more database operations (read/write) treated as an indivisible (atomic) unit.

ACID Properties (Foundation of Transactions)

PropertyFull FormDescriptionHow DBMS Ensures It
AAtomicityAll operations succeed or none are appliedUndo/Redo logs, rollback on failure
CConsistencyDatabase moves from one consistent state to anotherConstraints, triggers, application logic
IIsolationConcurrent transactions appear to execute seriallyLocks, multiversion concurrency control
DDurabilityCommitted changes survive failuresWrite-ahead logging (WAL), force to disk

Transaction States

  • Active → executing operations
  • Partially committed → final commit statement executed
  • Committed → successfully completed
  • Failed → cannot proceed (error detected)
  • Aborted → rolled back, effects undone
-- Simple transaction example (SQL)
BEGIN TRANSACTION;
    UPDATE Account SET balance = balance - 1000 WHERE acc_id = 101;
    UPDATE Account SET balance = balance + 1000 WHERE acc_id = 202;
    INSERT INTO TransactionLog (from_acc, to_acc, amount, t_date)
    VALUES (101, 202, 1000, CURRENT_DATE);
COMMIT;
-- or ROLLBACK if error occurs
Real-world example: Bank fund transfer must be atomic — deduct from A and credit to B must both happen or neither.

8.2 Concept of Serializability

Serializability ensures that concurrent execution of transactions produces the same result as some serial (one-after-another) execution of the same transactions.

Types of Serializability

  • Conflict Serializability: Schedule is conflict equivalent to some serial schedule
    • Conflict: two operations from different transactions on same data item, at least one is write
    • Read-Write, Write-Read, Write-Write conflicts
  • View Serializability: Stronger but harder to check (same initial read, same final writes, same reads-from relation)

Precedence Graph (Conflict Graph) Test

  • Nodes = transactions
  • Directed edge Ti → Tj if operation of Ti precedes and conflicts with operation of Tj
  • Schedule is conflict serializable ⇔ precedence graph is acyclic

// Example precedence graph
T1: read(A) → write(A)
T2: read(A) → write(B)
T3: write(A)

Conflicts:
T1 → T3  (write(A) after read(A) by T1)
T2 → T3  (write(A) after read(A) by T2)

Graph: T1 → T3 ← T2   → acyclic → serializable
Important: Conflict serializability is sufficient for most practical purposes and is easier to enforce via locking.

8.3 Concurrent Execution

Concurrency allows multiple transactions to execute simultaneously to improve throughput and resource utilization.

8.4 Lock-based Concurrency Control

Locking is the most common mechanism to achieve isolation and serializability.

Types of Locks

  • Shared Lock (S / Read Lock): Multiple transactions can hold S on same item (for reading)
  • Exclusive Lock (X / Write Lock): Only one transaction can hold X (for writing); blocks all others

Lock Compatibility Matrix

Requested \ HeldNoneSX
S (Shared)YesYesNo
X (Exclusive)YesNoNo

Lock Granularity

  • Row-level (fine) → high concurrency, more overhead
  • Table-level (coarse) → low concurrency, less overhead
  • Page-level → compromise (common in many DBMS)

8.5 2PL and Strict 2PL

Two-Phase Locking (2PL) Protocol

Transaction is divided into two phases:

  1. Growing Phase: Acquire locks (S or X), no release
  2. Shrinking Phase: Release locks, no new locks

Theorem: If all transactions follow 2PL → schedule is conflict serializable

Strict 2PL (Stronger & Most Common)

  • All exclusive locks held until commit/abort
  • Prevents cascading rollback (dirty read problem solved)
  • Most DBMS use Strict 2PL by default

// Strict 2PL behavior
T1: lock-X(A) → write(A) → lock-S(B) → read(B) → commit → unlock(A,B)
No release before commit
Deadlock: Cycle in wait-for graph (T1 waits for T2’s lock, T2 waits for T1’s). DBMS detects & resolves via timeout or victim selection.

8.6 Timestamp Concept

Timestamp-based concurrency control assigns a unique timestamp to each transaction (usually system time or logical counter) and orders conflicting operations based on timestamps.

Basic Timestamp Ordering (TO) Protocol

  • Each data item has:
    • Read-TS(Q) – largest TS of transaction that read Q
    • Write-TS(Q) – largest TS of transaction that wrote Q
  • For read(Q) by Ti:
    • If TS(Ti) < Write-TS(Q) → reject (Thomas’ write rule may ignore late writes)
  • For write(Q) by Ti:
    • If TS(Ti) < Read-TS(Q) or TS(Ti) < Write-TS(Q) → reject

Thomas’ Write Rule

If TS(Ti) < Write-TS(Q), ignore the write (late write) → avoids cascading rollback.

Multiversion Timestamp Ordering (MVTO)

  • Maintain multiple versions of each data item
  • Reader gets most recent version written by transaction with TS ≤ reader’s TS
  • Very good for read-heavy workloads (PostgreSQL MVCC uses similar concept)
