2.1 Conceptual, Logical and Physical Model

Database design follows a three-level schema architecture (ANSI/SPARC) that separates concerns across different abstraction levels.

LevelPurposeFocusWho Uses ItTypical ArtifactsIndependence
Conceptual ModelWhat data is needed & how entities relate (real-world view)Business concepts, entities, relationships, no implementation detailsBusiness analysts, domain experts, DB designersER Diagram, Conceptual SchemaLogical & Physical independent
Logical ModelHow data is structured logically (DBMS-independent)Tables, columns, keys, constraints, normalization (usually relational)Database designers, developersRelational Schema, Logical ERD (with tables & keys)Physical independent
Physical ModelHow data is stored & accessed on hardwareFile structures, indexes, partitions, storage engines, compressionDBAs, performance engineersPhysical DDL, Index definitions, Storage parameters—

Mapping Example (University Database):

  • Conceptual: Student enrolls in Course taught by Professor
  • Logical: Tables: Student(sid, name, …), Course(cid, title, …), Enrollment(sid, cid, grade)
  • Physical: B-tree index on Enrollment(sid, cid), partitioning by semester, InnoDB tablespace
Key Benefit of Separation: Change storage engine (InnoDB → MyRocks) or add index → only physical level changes. Add new attribute to Course → only logical & physical affected (if view exists, external level may remain unchanged).

2.2 Introduction to Entity Relationship Model

The Entity-Relationship (ER) Model, proposed by Peter Chen in 1976, is the most widely used **conceptual modeling** tool for databases.

Purpose: Represent real-world data requirements in a high-level, user-understandable way before translating to any specific DBMS model (usually relational).

Basic Components:

  • Entity (thing of interest)
  • Attribute (property of entity)
  • Relationship (association between entities)

Extended ER (EER) additions: Specialization/Generalization, Categories, Aggregation.

Strengths of ER Model: Intuitive for non-technical stakeholders, supports top-down design, easy translation to relational schema.

2.3 Entities Type

  • Strong / Regular Entity: Has its own unique identifier (primary key). Exists independently.
    Example: Student, Book, Department
  • Weak / Dependent Entity: Cannot exist without an owner entity. Identifier includes owner’s key + partial key.
    Example: Dependent (of Employee), LoanPayment (of Loan), Section (of Course)

// Strong Entity
Student (sid PK, name, email)

// Weak Entity
Dependent (emp_id FK + dependent_name PK composite, relation, dob)
// Identifying relationship: Employee → Dependent (total participation, identifying)

2.4 Entity Set

An entity set (or entity type) is a collection of similar entities that share the same attributes.

  • Entity Set: The collection (table concept in relational terms)
  • Entity Instance / Entity: A single member of the set (row / tuple)

Examples:

  • Entity Set: STUDENT
  • Entity Instances: Ram Sharma (sid=075BCT001), Sita Gurung (sid=075BCT015)

Notation in ER Diagram: Rectangle for entity set

Types of Keys

  • Super Key: Any set of attributes that uniquely identifies an entity
  • Candidate Key: Minimal super key (no subset is super key)
  • Primary Key: Chosen candidate key
  • Composite Key: More than one attribute
  • Foreign Key: References primary key of another entity
  • Partial Key (Discriminator): Distinguishes weak entities within owner

2.6 Relationship Types and Sets

A relationship associates two or more entities.

  • Relationship Type: Schema level (e.g., "Enrolls", "WorksIn")
  • Relationship Set: Instance level (actual associations)

Degree of Relationship

  • Unary / Recursive (degree 1): Employee supervises Employee
  • Binary (degree 2): most common (Student enrolls Course)
  • Ternary (degree 3): Supplier provides Part to Project

Cardinality / Mapping Constraints

RatioSymbolDescriptionExample
One-to-One (1:1)1 — 1One entity relates to exactly onePerson has Passport
One-to-Many (1:N)1 — NOne relates to manyDepartment has Employees
Many-to-One (N:1)N — 1Many relate to oneStudents belong to Department
Many-to-Many (N:M)N — MMany to manyStudents enroll in Courses

Participation Constraints

  • Total Participation: Every entity must participate (double line)
  • Partial Participation: Optional (single line)

Recursive Relationship Example


Employee (eid, name, …)
Supervision (supervisor_eid → Employee, supervisee_eid → Employee)

2.7 E-R Diagrams

Graphical representation using standard Chen notation (most common in academia).

Basic Symbols (Chen Notation)

  • Entity Type: Rectangle
  • Weak Entity Type: Double rectangle
  • Attribute: Oval (key = underlined)
  • Multi-valued attribute: Double oval
  • Derived attribute: Dashed oval
  • Relationship Type: Diamond
  • Identifying relationship (for weak entity): Double diamond
  • Cardinality: 1, N, M near lines
  • Total participation: Double line from entity to relationship

Example: University Mini ERD (Textual Representation)


[Student] ---- enrolls ---- [Course]
  |  N                  M  |
  |                         |
 double line (total)     single line (partial)

[Department] ---- offers ---- [Course]
  |  1                    N  |

[Professor] ---- teaches ---- [Course]
  |  N                    1  |

[Student] ---- has ---- [Dependent]   ← weak entity
  |  1                    N  |        double rectangle + double diamond

Steps to Construct ER Diagram

  1. Identify entities and entity sets
  2. Identify attributes and classify them
  3. Determine keys (primary, partial)
  4. Identify relationships and their types
  5. Specify cardinality and participation constraints
  6. Handle weak entities and identifying relationships
  7. Refine with specialization/generalization if needed
Common Mistakes:
• Using relationships instead of attributes (e.g., "hasPhone" as relationship instead of multi-valued attribute)
• Forgetting total participation for weak entities
• Confusing cardinality direction
