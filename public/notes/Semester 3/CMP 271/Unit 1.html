1.1 History, Database System and Its Applications

Brief History of Database Systems

  • 1960s: File-based systems → Hierarchical (IMS by IBM, 1968) and Network (CODASYL) models.
  • 1970: E.F. Codd publishes "A Relational Model of Data for Large Shared Data Banks" → birth of relational model.
  • 1970s–80s: First relational DBMS: System R (IBM), INGRES (UC Berkeley), Oracle (1979 – first commercial RDBMS).
  • 1980s–90s: SQL standardization (ANSI/ISO), widespread adoption of RDBMS (Oracle, DB2, SQL Server, MySQL).
  • 2000s: Open-source rise (MySQL, PostgreSQL), NoSQL emergence (key-value, document, column-family, graph) for big data & scalability.
  • 2010s–present: Cloud databases (AWS RDS, Google Cloud Spanner, Azure Cosmos DB), NewSQL (CockroachDB, TiDB), Multimodel, AI-integrated DBMS.

What is a Database System?

A database is an organized collection of structured/related data, typically stored electronically in a computer system.

A Database Management System (DBMS) is software that enables users to define, create, maintain, and control access to the database.

1.2 Characteristics of DBMS

  • Data Independence: Physical & logical independence (detailed in 1.4)
  • Efficient Data Access: Indexing, query optimization
  • Data Integrity Constraints: Entity, Referential, Domain, Key constraints
  • Concurrent Access & Recovery: Transaction management, ACID properties
  • Reduced Application Development Time: Centralized control, standard interfaces (SQL)
  • Multiple User Interfaces: GUI tools, SQL, APIs, reporting tools
  • Security: Authentication, Authorization, Encryption, Audit trails
  • Data Consistency: Enforced via constraints and transactions
  • Data Atomicity: All-or-nothing transaction execution
  • Backup & Recovery: Automatic logging, point-in-time recovery
Comparison: File System vs DBMS
File systems → data redundancy, no concurrency control, no query language, difficult integrity enforcement.
DBMS → centralized control, minimal redundancy, powerful query capability, strong consistency & recovery.

1.4 Data Abstraction and Independence

Levels of Data Abstraction (ANSI/SPARC 3-level Architecture)

  1. External / View Level: User-specific views (what users see). Multiple views from same data.
  2. Conceptual / Logical Level: Community view – overall logical structure (entities, relationships, constraints). Independent of physical storage.
  3. Internal / Physical Level: How data is actually stored (files, blocks, indexes, compression).

Data Independence

  • Physical Data Independence: Changes in storage (file structures, indexing) do not affect conceptual schema or applications.
  • Logical Data Independence: Changes in conceptual schema (add column, new entity) do not require changes in external views or applications (if properly designed).
Benefit: Applications remain stable even when database grows or technology changes.

1.5 Schemas and Instances

  • Schema: Description / blueprint of the database (structure). Relatively static.
    • Database schema = tables, columns, data types, constraints, relationships
    • Also called intension
  • Instance: Actual data stored at a particular moment (snapshot). Changes frequently.
    • Also called extension / state

// Example
Schema (DDL):
CREATE TABLE Student (
  sid INT PRIMARY KEY,
  name VARCHAR(50) NOT NULL,
  dob DATE,
  gpa DECIMAL(3,2)
);

Instance (DML - snapshot at time T):
sid | name          | dob        | gpa
1   | Ram Sharma    | 2003-05-12 | 3.85
2   | Sita Gurung   | 2004-01-30 | 3.60

1.6 Database Manager and Users

Database Manager / Administrator (DBA) Responsibilities

  • Schema definition & modification
  • Storage structure & access method decisions
  • Security & authorization management
  • Performance tuning (indexes, query optimization)
  • Backup, recovery, disaster planning
  • Monitoring usage & resolving concurrency issues
