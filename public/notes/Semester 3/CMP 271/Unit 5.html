Unit 5: Query Processing [3 Hrs.]

5.1 Introduction to Query Processing

Query processing is the sequence of activities performed by the DBMS to execute a user query and deliver the result.

High-level steps in query processing:

  1. Parsing & Translation: Validate syntax, convert SQL → internal representation (parse tree / relational algebra tree)
  2. Optimization: Find the most efficient execution plan among many possible equivalent plans
  3. Evaluation / Execution: Execute the chosen plan using access methods, join algorithms, etc.
  4. Result Delivery: Return results to user/application

Query Processor Components:

  • Parser / Validator
  • Query Rewriter / Normalizer
  • Query Optimizer (cost-based / heuristic)
  • Query Executor / Engine
  • Access Methods (index scan, table scan, join methods)
Goal of query processing: Minimize response time and resource consumption (I/O, CPU, memory) while producing correct results.

5.2 Query Cost Estimation

Cost-based optimizers estimate the cost of each possible execution plan and choose the one with the lowest estimated cost.

Cost Components (most DBMS use I/O cost as dominant)

  • I/O cost: Number of disk block accesses (seek + transfer time)
  • CPU cost: Processing time for comparisons, hashing, sorting
  • Memory cost: Buffer usage (less important in modern systems)
  • Network cost: In distributed databases

Important Statistics Maintained by DBMS (Statistics / Metadata)

  • Number of tuples in relation (cardinality) – |r|
  • Number of blocks – b(r)
  • Number of distinct values for attribute A – V(A, r)
  • Selection cardinality (selectivity) – number of tuples satisfying condition
  • Histogram / distribution information (for better estimation)
  • Index information (height, fan-out, clustering factor)

Cost Estimation Formulas (Simplified)

-- Table scan cost
Cost = b(R)   // number of blocks in R

-- Index lookup (non-clustered / secondary index)
Cost ≈ height(index) + number of matching tuples

-- Selection selectivity
σ_p(R) ≈ |R| × (1 / V(A,R))   for equality condition A = c

-- Join selectivity (natural join R ⋈ S on common attribute)
Selectivity ≈ 1 / max(V(A,R), V(A,S))
Challenge: Accurate statistics are crucial. Outdated statistics → poor plans. Most DBMS provide commands like ANALYZE TABLE / UPDATE STATISTICS.

5.3 Query Operations

DBMS implements several low-level operations efficiently:

Selection (σ) Implementation

  • File scan / linear search
  • Index scan (B⁺ tree, hash index)
  • Index + lookup (covering index if possible)

Other Operations

  • Projection (π) – usually cheap (drop columns)
  • Union, Intersection, Difference – often use sort-merge or hash
  • Sort (ORDER BY) – external merge sort if data > memory
  • Duplicate elimination (DISTINCT) – sort or hash
  • Grouping & Aggregation – hash or sort based

5.4 Evaluation of Expressions

Complex queries are translated into relational algebra trees (or query trees).

Evaluation techniques:

  • Materialization (Pipelined / On-the-fly): Produce tuples as soon as possible → better for large intermediate results
  • Materialization: Store intermediate result on disk → used when memory is limited

Example Relational Algebra Tree:

Query:
SELECT s.name, c.title
FROM Student s, Enrollment e, Course c
WHERE s.sid = e.sid AND e.cid = c.cid AND s.dept = 'CS' AND e.grade = 'A';

Relational Algebra:
π_name,title ( σ_dept='CS' ∧ grade='A' (Student ⋈ Enrollment ⋈ Course) )

Possible execution orders:

  • Apply selections first → reduce cardinality early (pushdown)
  • Join smallest relations first
  • Use indexes for selections and joins

5.5 Query Optimization

Query optimization finds the lowest-cost equivalent execution plan.

Types of Optimizers

  • Rule-based / Heuristic: Apply fixed transformation rules (MySQL older versions)
  • Cost-based: Enumerate plans, estimate costs using statistics (most modern DBMS: PostgreSQL, Oracle, SQL Server)

Important Optimization Techniques

  • Selection & Projection Pushdown: Apply σ and π as early as possible
  • Join Ordering: Choose best order (dynamic programming or greedy)
  • Predicate Pushdown: Move conditions into subqueries/views
  • Subquery Unnesting / Flattening: Convert correlated subqueries to joins
  • Index Selection: Use covering indexes, clustered indexes
  • Materialized Views & Query Rewrite: Use pre-computed results
  • Parallel Execution: In distributed / multi-core systems

EXPLAIN / EXPLAIN ANALYZE

-- PostgreSQL / MySQL
EXPLAIN ANALYZE
SELECT * FROM Student s JOIN Enrollment e ON s.sid = e.sid
WHERE s.dept = 'CS';

-- Shows: seq scan vs index scan, join type, estimated vs actual rows, cost
Key Insight: Optimizer is only as good as its statistics and cost model. Poor cardinality estimation → wrong plan → slow queries. Always run ANALYZE / UPDATE STATISTICS after large data changes.
