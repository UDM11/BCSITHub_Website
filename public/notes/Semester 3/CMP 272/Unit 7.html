Unit 7: Case Study and Project [4 Hrs.]

7.1 Apply OOAD Principles to a Real-World Case Study

Chosen Case Study: Online Library Management System (OLMS)

A modern web-based library system that allows members to search, reserve, borrow, renew, and return books digitally, while librarians manage inventory, members, fines, and reports. The system supports multiple user roles, notifications, and integration with payment gateways for fines.

Stakeholders & Actors

  • Member (Student, Faculty, Guest) – searches, reserves, borrows, renews, returns, pays fines
  • Librarian – manages books, members, loans, reservations, generates reports
  • Admin – manages user accounts, system settings, categories
  • Payment Gateway (external system) – processes fine payments
  • Notification Service (email/SMS) – overdue reminders, reservation ready

High-Level Functional Requirements (Use Cases Summary)

  1. Register / Login / Logout
  2. Search Books (by title, author, ISBN, category)
  3. View Book Details & Availability
  4. Reserve Book
  5. Borrow Book (if available)
  6. Renew Loan (if eligible)
  7. Return Book
  8. Pay Fine
  9. Manage Books (Add, Update, Remove) – Librarian
  10. Manage Members (Activate, Suspend) – Librarian/Admin
  11. Generate Reports (Overdue, Popular Books) – Librarian
  12. Send Overdue/Reservation Notifications

Non-Functional Requirements

  • Concurrency: Multiple members borrowing/reserving simultaneously
  • Security: Role-based access, password hashing
  • Usability: Responsive web interface
  • Reliability: Transactional integrity (loan + status update atomic)
  • Performance: Search < 2 seconds, report generation < 5 seconds

Applying OOAD Principles & Patterns

Concept / PrincipleApplication in OLMS
GRASP – Information ExpertBook knows its own availability & loan history
GRASP – CreatorLoan creates LoanItem when borrowing
GRASP – ControllerLibraryController or UseCase controllers (BorrowBookController)
GRASP – Low Coupling / High CohesionSeparate domain (Book, Loan), service (LoanService), repository layers
SOLID – SRPLoanService only handles borrowing/renewal/return logic
SOLID – OCPNotificationStrategy (Email, SMS) – add new types without changing core
SOLID – DIPLoanService depends on INotificationService interface
GoF – SingletonConfigurationManager, NotificationQueue
GoF – StrategyFineCalculationStrategy (Student vs Faculty different rules)
GoF – ObserverLoan observes → notifies Member & Librarian on overdue
GoF – FacadeLibraryFacade hides complexity of multiple services
GoF – AdapterPaymentAdapter to integrate different gateways (Stripe, eSewa)

3. Sequence Diagram Example – Borrow Book (Happy Path)


Member -> LibraryController: requestBorrow(isbn)
LibraryController -> BookRepository: findByIsbn(isbn)
BookRepository --> LibraryController: book
LibraryController -> Book: isAvailable()
alt available
  Book --> LibraryController: true
  create Loan
  LibraryController -> Loan: new Loan(member, book)
  Book -> Book: borrowCopy()  // decrement copies
  LibraryController -> LoanRepository: save(loan)
  LibraryController -> NotificationService: send("Book borrowed", member)
  LibraryController --> Member: success + dueDate
else
  LibraryController -> ReservationService: createReservation(member, book)
  LibraryController --> Member: reserved
end

Returned  --> [*]

7.3 Class Presentations and Discussions

Project Deliverables (Typical):

  • Requirements Specification (use case descriptions + NFRs)
  • Complete UML Diagrams:
    • Use Case Diagram
    • Domain Model (Class Diagram – conceptual)
    • Sequence / Communication Diagrams (2–3 critical scenarios)
    • Activity Diagram (e.g., Borrow Book workflow)
    • State Machine (Loan, Reservation, Member Account)
    • Design Class Diagram (with patterns applied)
    • Package Diagram (layers: presentation, application, domain, infrastructure)
  • Design Decisions Document:
    • Which GRASP principles applied & why
    • SOLID compliance examples
    • GoF patterns used + justification
    • Trade-offs (e.g., Singleton vs DI, Strategy vs if-else)
  • Prototype / Partial Implementation (optional – 1–2 use cases)
  • Presentation Slides (15–20 min):
    • Problem statement & scope
    • Actors & use cases
    • Key diagrams walkthrough
    • Patterns applied with code snippets
    • Lessons learned & improvements

Discussion Points for Class / Viva:

  • Why choose composition over deep inheritance?
  • How did you handle concurrency (e.g., two members borrowing last copy)?
  • Where did you apply OCP? Could you extend for new fine rules easily?
  • What would break LSP if you added a restricted member type?
  • Alternative patterns you considered but rejected (and why)?
  • How would you scale this system to 100,000+ members?
Tip for Students: Document every major design decision with justification (GRASP, SOLID, pattern name). This is what examiners value most in case study evaluations.
