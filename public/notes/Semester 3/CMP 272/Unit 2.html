Unit 2: Requirement Elicitation and Analysis [7 Hrs.]

2.1 Requirement Gathering

Requirement gathering (also called requirements engineering) is the foundational phase in OOAD where we discover, document, and refine what the system must do and how well it must perform. In object-oriented approaches, this phase heavily emphasizes identifying actors, use cases, and domain concepts that will later map to classes and objects.

2.1.1 Requirement Fundamentals

Requirements are statements about system capabilities, constraints, and qualities. They are classified into:

  • Functional Requirements: What the system shall do (e.g., "The system shall allow users to borrow books"). Expressed via use cases and scenarios in OOAD.
  • Non-Functional Requirements (NFRs): How well the system performs (qualities/attributes):
    • Performance (response time, throughput)
    • Usability (ease of use, learnability)
    • Reliability (availability, fault tolerance)
    • Security (authentication, confidentiality)
    • Scalability, Maintainability, Portability, etc.
  • Constraints: Restrictions (technology, budget, standards, legal).
  • Business Rules: Policies or conditions (e.g., "Fine = 10 NPR per day after due date").

Qualities of Good Requirements (SMART / INVEST-like): Specific, Measurable, Achievable, Relevant, Traceable, Complete, Consistent, Unambiguous, Verifiable, Correct, Feasible.

2.1.2 Requirement Elicitation Techniques

Elicitation is the process of discovering requirements from stakeholders. Common techniques include:

  1. Interviews: Structured (questionnaire-based), semi-structured, or unstructured. One-on-one or group. Best for deep insights but time-consuming.
  2. Questionnaires / Surveys: Useful for large groups, quantitative data. Closed/open questions. Risk: low response rate or misunderstanding.
  3. Workshops / JAD (Joint Application Design): Facilitated meetings with stakeholders, developers, users. Brainstorming, consensus building. Effective for conflict resolution.
  4. Observation / Shadowing: Watching users perform tasks in real environment. Reveals unspoken needs and inefficiencies (ethnography-inspired).
  5. Document Analysis: Reviewing existing manuals, forms, legacy systems, business rules, regulations.
  6. Brainstorming: Group idea generation without criticism. Good for innovative features.
  7. Prototyping: Throwaway or evolutionary prototypes to clarify vague requirements. Users see mock-ups → feedback loop.
  8. Domain Analysis: Studying similar systems or standards in the domain.
  9. Focus Groups: Guided discussions with representative users.
  10. Role-Playing / Scenarios: Acting out interactions to uncover edge cases.

Challenges in Elicitation: Tacit knowledge, conflicting stakeholder views, changing requirements, resistance to change, language barriers.

2.1.3 Requirement Analysis

After elicitation, analyze requirements for conflicts, completeness, feasibility. In OOAD:

  • Identify domain objects and relationships (start class diagram).
  • Prioritize requirements (MoSCoW: Must, Should, Could, Won't).
  • Model behavior (use cases, activity flows).
  • Resolve ambiguities and inconsistencies.

2.1.4 Requirement Specification

Document requirements formally. In OOAD, use:

  • Use Case Descriptions (template: Name, Actors, Preconditions, Postconditions, Main Flow, Alternative Flows, Exceptions).
  • UML Diagrams (use case, class, activity).
  • Supplementary Specification (NFRs, glossary, assumptions).

// Example Use Case Template (Textual)
USE CASE: Borrow Book
Actors: Member, Librarian
Preconditions: Member is registered, Book is available
Main Success Scenario:
  1. Member requests to borrow book.
  2. Librarian scans ISBN.
  3. System checks availability and due date.
  4. System records loan.
  5. System issues book.
Extensions:
  3a. Book unavailable → Alternative: Reserve book.
Exceptions:
  Member has overdue books → Deny borrowing.

2.1.5 Requirement Validation

Ensure requirements are correct, complete, and satisfy stakeholders. Techniques:

  • Reviews / Inspections
  • Prototyping walkthroughs
  • Test case derivation
  • Stakeholder sign-off
  • Traceability matrix (requirements → use cases → design → tests)

2.4 Use Cases, Scenarios and Actors

Use cases capture functional requirements from the user's perspective in OOAD.

  • Actor: External entity interacting with the system (human user, another system, timer, device). Represented as stick figure in UML. E.g., Member, Librarian, Payment Gateway.
  • Use Case: A single, cohesive functionality providing value to an actor. Named as verb + noun (e.g., Borrow Book, Process Payment). Oval in UML.
  • Scenario: A specific path through a use case (instance). Includes:
    • Main success scenario (happy path)
    • Alternative flows
    • Exception flows

Relationships in Use Case Diagrams: Include (mandatory sub-use case), Extend (optional extension), Generalization (inheritance between actors/use cases).


// Textual representation of Use Case Diagram for Library
Actors: Member, Librarian, Admin
Use Cases:
  - Register Member (by Librarian)
  - Borrow Book (by Member) -- includes --> Check Availability
  - Return Book (by Member)
  - Pay Fine (by Member) -- extends --> Borrow Book (if overdue)

2.5 Overview of the Unified Modeling Language: UML Fundamentals and Notations

UML (Unified Modeling Language) is a standardized visual modeling language for specifying, visualizing, constructing, and documenting software systems. Version 2.x is current. It has 14 diagram types divided into Structure (static) and Behavior (dynamic).

2.5.1 Behavior Diagram: Use Case Diagram and Activity Diagram

Use Case Diagram: Shows system functionality from external view. Elements: Actors, Use Cases, System boundary, relationships (include, extend, generalization).

Activity Diagram: Flowchart-like. Models workflow, business processes, use case details. Elements: Action, Initial/Final Node, Decision Node (diamond), Merge, Fork/Join (parallel), Swimlanes (partitions by actor/responsibility).


// Activity Diagram for Borrow Book (textual)
[Initial] --> [Member requests book] --> Decision: Available?
  Yes --> [Update loan record] --> [Issue book] --> [Final]
  No  --> [Notify unavailable] --> [Final]
Swimlanes: Member | System | Librarian

2.5.2 Interaction Diagram: Sequence and Communication Diagram

Sequence Diagram: Shows object interactions over time. Vertical lifelines (objects/actors), horizontal messages (synchronous/async), activation bars, loops, alternatives (alt), options (opt).

Communication Diagram (Collaboration): Emphasizes structure (links) over time. Numbers messages in sequence. Good for showing object relationships.


// Sequence Diagram textual (Borrow Book)
Member    -> Librarian: request book
Librarian -> System: scan ISBN
System    -> System: check availability
alt available
  System  -> Librarian: confirm
  Librarian -> Member: issue book
else
  Librarian -> Member: notify unavailable
end

2.5.3 Structure Diagram: Class, Package, and Component Diagram

Class Diagram: Static structure. Classes, attributes, operations, relationships (association, aggregation, composition, inheritance, dependency).

Package Diagram: Organizes elements into namespaces. Shows dependencies between packages (import, merge).

Component Diagram: High-level software components (subsystems, libraries), interfaces, provided/required ports.
