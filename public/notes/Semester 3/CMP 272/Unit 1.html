Unit 1: Introduction to Object-Oriented Analysis and Design [4 Hrs.]

1.1 Class, Objects, Attributes, Methods

Object-Oriented Analysis and Design (OOAD) is a software engineering methodology that models complex systems using objects that represent real-world or conceptual entities. It combines Object-Oriented Analysis (OOA) — understanding and specifying requirements through objects — and Object-Oriented Design (OOD) — structuring the solution with classes, relationships, and interactions.

  • Class: A blueprint or template that defines the common structure (attributes) and behavior (methods) for a group of similar objects. It describes what objects of that type look like and can do. Classes promote reusability and organization.
  • Object: A concrete instance of a class — a runtime entity with its own identity, state (attribute values), and behavior. Every object has a unique identity even if two objects have identical attribute values.
  • Attributes (Properties / Instance Variables / Data Members): Represent the state or characteristics of an object. They hold data values that can change during the object's lifetime.
  • Methods (Operations / Member Functions): Define the behavior or services an object can provide or receive. Methods manipulate attributes or perform actions.

Key Principles Illustrated:

  • Encapsulation: Bundling attributes and methods together inside a class, hiding internal details (data hiding via access modifiers like private/public).
  • Abstraction: Focusing on essential features while hiding complexity (class exposes only necessary methods).

Real-World Example – Library Management System:


// Class definition (Java-like pseudocode)
class Book {
    // Attributes (private for encapsulation)
    private String isbn;
    private String title;
    private String author;
    private int publicationYear;
    private boolean isAvailable;

    // Constructor (special method to initialize object)
    public Book(String isbn, String title, String author, int year) {
        this.isbn = isbn;
        this.title = title;
        this.author = author;
        this.publicationYear = year;
        this.isAvailable = true;
    }

    // Methods (behavior)
    public void borrowBook() {
        if (isAvailable) {
            isAvailable = false;
            System.out.println("Book '" + title + "' borrowed successfully.");
        } else {
            System.out.println("Book '" + title + "' is currently unavailable.");
        }
    }

    public void returnBook() {
        isAvailable = true;
        System.out.println("Book '" + title + "' returned.");
    }

    public String getDetails() {
        return "ISBN: " + isbn + ", Title: " + title + ", Author: " + author +
               ", Year: " + publicationYear + ", Available: " + isAvailable;
    }
}

// Creating objects
Book book1 = new Book("978-3-16-148410-0", "Clean Code", "Robert C. Martin", 2008);
Book book2 = new Book("978-0-201-63361-0", "Design Patterns", "Erich Gamma", 1994);

book1.borrowBook();
System.out.println(book1.getDetails());

Objects book1 and book2 are distinct instances with their own state, even though both are of type Book.

1.2 Relation between Objects: Association, Aggregation, and Composition Relationships with Examples

Objects rarely exist in isolation; they interact through relationships. In UML class diagrams, these are shown with lines connecting classes, often with symbols indicating multiplicity (e.g., 1..*, 0..1) and role names.

  • Association: A general, structural relationship indicating that objects of one class are connected to objects of another ("uses-a", "knows-a", or "related-to"). Objects have independent lifecycles. Can be unidirectional or bidirectional. Represented by a solid line in UML.
    • Example: A Student enrolls in a Course. A student can exist without courses, and courses exist without specific students.
    • Multiplicity: One student can enroll in many courses (1..*), one course has many students (*..*).
  • Aggregation: A "whole-part" or "has-a" relationship where the part can exist independently of the whole (weaker ownership). The part's lifecycle is not controlled by the whole. Represented by a hollow diamond on the whole side in UML.
    • Example: A Team has Players. If the team disbands, players can still exist and join other teams.
    • Common in collections like lists, arrays, or shared resources.
  • Composition: A stronger "whole-part" relationship where the part cannot exist without the whole (strong ownership, exclusive lifecycle). If the whole is destroyed, its parts are destroyed too. Represented by a filled diamond on the whole side in UML.
    • Example: A Car has an Engine. If the car is scrapped, the engine ceases to be part of any car (though physically it might be reused, conceptually it's destroyed with the car in the model).
    • Another: A Document has Paragraphs — paragraphs don't exist outside their document.

Comparison Table (Conceptual):

  • Association → Loose connection, independent lifecycles
  • Aggregation → "Has-a" weak, parts shared/independent
  • Composition → "Has-a" strong, parts owned/exclusive

// Association
class Student {
    Course[] enrolledCourses;  // Student knows Courses
}

class Course {
    Student[] enrolledStudents;  // Bidirectional possible
}

// Aggregation
class Team {
    List<Player> players = new ArrayList<>();  // Players can exist without Team
}

// Composition
class Car {
    private Engine engine = new Engine();  // Engine created & destroyed with Car
    private Wheel[] wheels = new Wheel[4];
    
    public Car() {
        for (int i = 0; i < 4; i++) wheels[i] = new Wheel();
    }
}

Common Misconception: Aggregation vs Composition is not just about physical containment but about lifecycle dependency and ownership responsibility in the software model.

1.3 Benefits and Challenges of OOP (and OOAD)

OOP/OOAD revolutionized software development by promoting modularity, reusability, and better modeling of real-world problems. However, it introduces complexity that must be managed carefully.

  • Benefits / Advantages:
    • Modularity & Maintainability: Code organized into self-contained classes/objects → easier debugging, updates, and team collaboration.
    • Reusability: Classes, inheritance, polymorphism, and libraries allow code reuse across projects (e.g., frameworks like Spring, .NET).
    • Real-World Mapping: Objects model entities (Person, Account, Order) → improves understanding, communication with stakeholders, and requirement analysis.
    • Encapsulation & Data Protection: Private attributes + public methods → reduces bugs from invalid data states.
    • Polymorphism & Flexibility: Same method name behaves differently (e.g., draw() in Circle, Rectangle) → extensible systems without modifying existing code.
    • Inheritance & Hierarchy: Promotes DRY (Don't Repeat Yourself) principle and hierarchical organization.
    • Scalability: Supports large systems (enterprise software, games, simulations) via design patterns (Singleton, Factory, Observer, etc.).
    • Parallel Development: Teams can work on different classes/objects independently.
  • Challenges / Disadvantages:
    • Steep Learning Curve: Requires deep understanding of abstraction, encapsulation, inheritance, polymorphism, SOLID principles, and design patterns.
    • Performance Overhead: Object creation, virtual method calls, garbage collection → slower than procedural code in some cases (critical in real-time/embedded systems).
    • Over-Engineering Risk: Excessive abstraction, deep inheritance trees ("yo-yo problem"), or god classes → hard-to-maintain "spaghetti" code.
    • Design Complexity: Poorly chosen relationships (tight coupling, fragile base class problem) lead to ripple effects during changes.
    • Higher Memory Usage: Objects carry overhead (references, vtables); many small objects can fragment memory.
    • Debugging Difficulty: Polymorphic behavior and dynamic dispatch make tracing harder than linear procedural code.
    • Initial Development Time: More upfront design effort compared to simple scripts or procedural programs.

When to Use OOP/OOAD: Ideal for complex, evolving systems (banking, e-commerce, games, GUI apps). May be overkill for small scripts or performance-critical code (use procedural/C-style when appropriate).
