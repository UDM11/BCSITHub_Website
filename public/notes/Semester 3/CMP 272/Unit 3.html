Unit 3: Object Oriented Analysis [6 Hrs.]

3.1 Domain Modeling and Concept Modeling Relationships

Domain modeling is the most critical activity in Object-Oriented Analysis (OOA). It involves creating a conceptual model of the real-world problem domain — independent of implementation details — that captures important entities, their attributes, and how they relate to each other.

Objectives of Domain Modeling

  • Understand the problem domain deeply from the business/user perspective.
  • Identify core concepts (nouns → candidate classes) and relationships.
  • Create a stable foundation for use-case realization and system design.
  • Provide a shared vocabulary between stakeholders, analysts, and developers.
  • Avoid premature design decisions (implementation classes come later).

Key Elements in Domain Modeling

  • Conceptual Classes: Important "things" in the domain (nouns or noun phrases). E.g., Book, Member, Loan, Reservation, Fine.
  • Attributes: Characteristics of classes (simple data types or other classes). E.g., title: String, dueDate: Date, fineAmount: float.
  • Associations / Relationships: How classes are connected. Named with verb phrases. Multiplicity must be specified (1, *, 0..1, 1..*, etc.).
  • Generalization / Specialization: Is-a relationships (inheritance). E.g., Student is-a Member.

Common Types of Relationships in Domain Models

Relationship TypeUML SymbolDescriptionExample
Associationsolid lineGeneral structural link between classesMember borrows Book
Aggregationhollow diamondWhole-part, part can exist independentlyShoppingCart has Items

Real-world Domain Model Example: Library Management System


// Textual UML-like Domain Model (Conceptual Class Diagram)
class Library {
  name: String
}

class Member {
  memberId: String
  name: String
  email: String
  registrationDate: Date
  membershipType: String  // Regular, Premium, Student
}

class Book {
  isbn: String
  title: String
  author: String
  publicationYear: int
  category: String
  totalCopies: int
  availableCopies: int
}

class Loan {
  loanId: String
  borrowDate: Date
  dueDate: Date
  returnDate: Date
  fineAmount: float
}

class Reservation {
  reservationId: String
  reservationDate: Date
  status: String  // Pending, Fulfilled, Cancelled
}

// Relationships
Library "1" -- "0..*" Member     : registers >
Library "1" -- "0..*" Book       : contains >
Member  "1" -- "0..*" Loan       : has >
Book    "1" -- "0..*" Loan       : is borrowed in >
Member  "1" -- "0..*" Reservation: makes >
Book    "1" -- "0..*" Reservation: is reserved for >
Best Practice Tip: Use noun phrase identification from requirements/use cases to find candidate classes. Apply filters: eliminate redundant, external, or implementation-specific terms.

3.2 Identifying Classes, Attributes, Methods

From the domain model and use cases, we refine conceptual classes into more detailed analysis classes. We also begin identifying responsibilities (methods) using CRC cards (Class-Responsibility-Collaboration) or responsibility-driven design.

Steps to Identify Classes, Attributes, Methods

  1. Noun Phrase Identification: Scan requirements and use-case descriptions for nouns/noun phrases → candidate classes.
  2. Apply Filters: Remove: synonyms, external actors, attributes disguised as classes, implementation terms (e.g., database, button).
  3. Attribute Identification: Look for adjectives, possessive phrases, or data mentioned with the noun (e.g., "book title" → title attribute of Book).
  4. Responsibility Identification (Methods):
    • What does this class know? (information-related → getters, calculations)
    • What can this class do? (behavior → actions, state changes)
    • Who initiates the behavior? (collaboration)
  5. CRC Card Technique: For each class: write Class name, Responsibilities (what it does/knows), Collaborators (other classes it needs).

Example: Online Shopping System – Class Identification & Responsibilities


// Candidate Classes after filtering
class Customer
class Product
class ShoppingCart
class Order
class OrderItem
class Payment
class Address
class Category

// Detailed with Attributes & Methods (Analysis Level)

class Customer {
  - customerId: String
  - name: String
  - email: String
  - phone: String
  - addresses: List<Address>
  + register()
  + login()
  + addToCart(Product, quantity)
  + placeOrder()
  + viewOrderHistory()
}

class ShoppingCart {
  - cartId: String
  - customer: Customer
  - items: List<CartItem>
  + addItem(Product, quantity)
  + removeItem(Product)
  + updateQuantity(Product, newQty)
  + calculateTotal()
  + clear()
}

class Order {
  - orderId: String
  - orderDate: Date
  - status: String  // Pending, Paid, Shipped, Delivered, Cancelled
  - totalAmount: double
  - shippingAddress: Address
  - payment: Payment
  - items: List<OrderItem>
  + createFromCart(ShoppingCart)
  + applyDiscount(Discount)
  + confirmPayment()
  + cancel()
  + getStatus()
}

Common Mistakes to Avoid:

  • God classes (too many responsibilities)
  • Modeling database tables instead of domain concepts
  • Confusing attributes with classes (e.g., Address should be class, not just String in Customer)
  • Ignoring multiplicity and direction of associations

3.3 Behavior Modeling using State Diagrams and Activity Diagrams

While class/structural diagrams show "what exists", behavior diagrams show "what happens over time".

State Machine Diagram (Statechart Diagram)

Models lifecycle of an object — states it can be in and transitions triggered by events, guarded by conditions, with possible effects.

  • State: Condition/situation during lifetime (rounded rectangle)
  • Transition: Arrow showing change (event [guard] / action)
  • Initial / Final Pseudostate
  • Composite / Submachine states for complex lifecycles

Example: State Diagram for Order in E-commerce


// Textual State Machine Diagram - Order Lifecycle
[*] --> Pending : create order

state Pending {
  [*] --> WaitingForPayment
  WaitingForPayment --> Paid     : payment received
  WaitingForPayment --> Cancelled: cancel requested [within 30 min]
}

state Paid {
  [*] --> Processing
  Processing --> Shipped     : items dispatched
  Processing --> Cancelled   : cancel after payment [refund initiated]
}

state Shipped {
  [*] --> InTransit
  InTransit --> Delivered   : delivery confirmed
  InTransit --> Returned    : customer returns item
}

Delivered --> [*] : completed
Cancelled --> [*] : order terminated
Returned  --> [*] : refund processed

Activity Diagram

Models workflow, business processes, or detailed use-case realization. More flowchart-like than state diagrams.

  • Action nodes, control flows, decision nodes, fork/join (parallelism), swimlanes (responsibility partitions)
  • Often used to detail main/alternative flows of use cases

Example: Activity Diagram for Place Order Use Case (with Swimlanes)


// Textual Activity Diagram - Place Order
partition Customer {
  [Browse products] --> [Add to cart] --> [View cart] --> Decision: Proceed to checkout?
    Yes --> [Enter shipping address]
    No  --> [Continue shopping] --> [Browse products]
}

partition System {
  [Validate address] --> [Calculate total + tax + shipping]
  --> [Display order summary]
}

partition Customer {
  [Confirm order] --> [Choose payment method]
}

partition Payment Gateway {
  fork
    --> [Process Credit Card]
    --> [Process Digital Wallet]
  join
}

partition System {
  Decision: Payment successful?
    Yes --> [Update order status to Paid] --> [Send confirmation email] --> [*]
    No  --> [Display error] --> [Return to payment selection]
}
State Diagram vs Activity Diagram – Quick Comparison:
State → Focuses on object lifecycle and state changes.
Activity → Focuses on workflow, sequence of actions, possibly across multiple objects/actors.
