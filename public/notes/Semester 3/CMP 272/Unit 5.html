5.1 General Responsibility Assignment Software Patterns (GRASP)

GRASP (General Responsibility Assignment Software Patterns) are nine fundamental guidelines (by Craig Larman) for assigning responsibilities to classes and objects during object-oriented design. They answer "Who should do what?" to achieve low coupling, high cohesion, and maintainable systems. GRASP patterns guide responsibility assignment in analysis-to-design transition.

5.1.1 Creator

Problem: Who should be responsible for creating instances of a class?
Solution: Assign creation responsibility to a class that aggregates, contains, records, closely uses, or has initializing data for the created class (BIE — Benefits, Initialization, Encapsulation).
Guidelines (in priority order): Class A creates B if A:

  1. contains / aggregates B (composition/aggregation)
  2. records instances of B
  3. closely uses B
  4. has initializing data for B

Example – Point-of-Sale (POS) System: Sale creates SaleLineItem (Sale contains/aggregates line items).


class Sale {
  private List<SaleLineItem> lineItems = new ArrayList<>();
  
  public SaleLineItem addLineItem(ProductDescription desc, int qty) {
    SaleLineItem newItem = new SaleLineItem(desc, qty);  // Sale is Creator
    lineItems.add(newItem);
    return newItem;
  }
}

5.1.2 Information Expert

Problem: What is the basic principle for assigning responsibilities?
Solution: Assign a responsibility to the class that has the most information needed to fulfill it ("Information Expert" = the expert on the data).
This promotes high cohesion and low coupling.

Example – Library System: Book knows its own availability → Book should have isAvailable() method, not Librarian.


class Book {
  private int availableCopies;
  
  public boolean isAvailable() {
    return availableCopies > 0;  // Information Expert: Book knows its state
  }
  
  public void borrow() {
    if (isAvailable()) availableCopies--;
  }
}

5.1.3 Controller

Problem: Who should handle system events (use case realization)?
Solution: Assign to a non-UI class representing the overall system, a use case, or a root object (Facade-like).
Types: Use-case controller (one per use case) or facade controller.

Example – POS: Register (or POSSystem) acts as Controller for "makeSale" use case.

5.1.4 Low Coupling

Problem: How to support low dependency, impact of change, and increased reuse?
Solution: Assign responsibilities so coupling between classes remains low (few dependencies).
Coupling types: High (bad) vs Low (good). Prefer data coupling over stamp/content coupling.

5.1.5 High Cohesion

Problem: How to keep objects focused, understandable, and manageable?
Solution: Assign responsibilities that are strongly related and cohesive (class does one thing well).

PrincipleGoalBenefitViolation Example
Low CouplingMinimize dependenciesEasier change, reuse, testOrder directly calls Database
High CohesionRelated responsibilitiesClear, maintainable classesOrder class handles UI, DB, business logic
GRASP Summary Tip: Use Information Expert first → then Creator, Controller → evaluate with Low Coupling / High Cohesion.

5.2 Designing for Visibility

Visibility (access modifiers) controls which classes can access attributes/methods. In UML and OO languages, visibility promotes encapsulation and reduces coupling.

VisibilityUML SymbolAccessible FromUse Case
Public+AnywhereInterface methods, API
Protected#Same class + subclassesTemplate methods, inheritance
Private-Only same classInternal state, helpers
Package (default)~Same packageInternal utilities in package

Example – UML & Code:


// UML-like
class Account {
  + getBalance(): double          // public
  # calculateInterest(): double   // protected
  - balance: double               // private
  ~ transactionLog: List          // package
}

// Java equivalent
public class Account {
  public double getBalance() { return balance; }
  protected double calculateInterest() { ... }
  private double balance;
  List<Transaction> transactionLog;  // package-private
}
Common Mistake: Making everything public → breaks encapsulation, high coupling.

5.3 SOLID Principles

SOLID (by Robert C. Martin / Uncle Bob) are five principles for better class design: maintainable, extensible, flexible code.

5.3.1 Single Responsibility Principle (SRP)

A class should have only one reason to change (one responsibility).

Violation Example: Order class handles calculation, persistence, printing.
Fix: Split into Order (business), OrderRepository (DB), OrderPrinter (UI/report).

5.3.2 Open Closed Principle (OCP)

Classes should be open for extension but closed for modification.

Use abstraction (interfaces/abstract classes) + polymorphism.
Example: PaymentProcessor with strategy pattern for CreditCard, PayPal, etc.


interface PaymentProcessor {
  void process(Payment payment);
}

class CreditCardProcessor implements PaymentProcessor { ... }
class PayPalProcessor implements PaymentProcessor { ... }

// Closed for modification, open via new implementations

5.3.3 Liskov Substitution Principle (LSP)

Subclasses must be substitutable for base classes without altering correctness.

Violation: Square extends Rectangle but overrides setWidth() to change height too.
Fix: Avoid mutable inheritance where behavior breaks contract; prefer composition.

5.3.4 Interface Segregation Principle (ISP)

Clients should not depend on interfaces they don't use (many small interfaces > one fat).

Example: Instead of huge IPrinter with scan/print/fax, have IPrinter, IScanner, IFax.

5.3.5 Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level; both depend on abstractions.
Abstractions should not depend on details; details depend on abstractions.

Example: Use interfaces (abstractions) + dependency injection.


// High-level depends on abstraction
class OrderService {
  private PaymentProcessor processor;  // interface
  
  public OrderService(PaymentProcessor p) { this.processor = p; }  // DIP + injection
}
SOLID Benefits: Easier testing, refactoring, extension; foundation for design patterns.
