Unit 4: Object-Oriented Modeling Using UML Notation [10 Hrs.]

4.1 Elaborating Use Cases and Use Case Diagram

Elaborating use cases involves detailing functional requirements beyond high-level overviews. This includes writing full specifications (name, actors, scope, level, preconditions, postconditions, main success scenario, extensions, exceptions) and refining the use case diagram with advanced relationships for better modularity and reuse.

4.1.1 Use Cases for Reuse

Reuse reduces redundancy by factoring common behavior into shared use cases. UML supports this via <>, <>, and generalization relationships.

  • <>: Mandatory inclusion of behavior (like subroutine call). The base use case is incomplete without it. Promotes reuse of common steps.
  • <>: Optional/conditional insertion at extension points. Base use case is complete; extension adds variant behavior.
  • Generalization: Inheritance between use cases or actors (child inherits and specializes parent behavior).

4.1.2 Use Cases and Components

Use cases can map to components in component diagrams (e.g., a use case realized by a subsystem or component). This bridges requirements to architecture.

4.1.3 Separation Variant Behavior

Use <> to separate optional/variant behavior (e.g., handling errors, alternatives) from the main flow, keeping base use cases clean.

4.1.4 Generalization

Use case generalization: Child use case inherits steps from parent and adds/specializes. Actor generalization: Specialized actors (e.g., PremiumMember) inherit from general (Member).

4.1.5 Actor as Classes

Actors can be modeled as classes in detailed design (e.g., external systems or roles with attributes/methods), especially in service-oriented or actor-generalization scenarios.

Example: Library Management System – Elaborated Use Case Diagram (Textual)


// Use Case Diagram with advanced relationships
Actors:
  - Member
  - Librarian
  - PremiumMember --|> Member   // Generalization (inherits from Member)

Use Cases:
  - Borrow Book (base)
  - Return Book
  - Pay Fine
  - Authenticate User <> Borrow Book   // Mandatory reuse
  - Check Overdue <> Borrow Book   // Optional variant at extension point "before issuing"
  - Renew Membership (specialized) --|> Manage Membership   // Generalization

System Boundary: Library System
Best Practice: Use <> for must-have shared steps (e.g., login), <> for conditionals (e.g., apply discount if premium), generalization for role hierarchies.

4.2 Logical Architecture and UML Package Diagram

Logical architecture organizes classes into layers/packages for separation of concerns. Package diagrams show namespaces, dependencies (<>, <>), and high-level structure.

4.2.1 Model-View Separation Principle

Model (domain/business logic) objects should not know about View (UI) objects directly. This enables independent development, testing, and UI changes without affecting core logic. Originates from MVC (Model knows neither View nor Controller directly in strict form).

Example Package Diagram – Online Banking (Textual)


// Package Diagram
package "Domain Model" {
  package "Entities" { Account, Transaction, Customer }
  package "Services" { AccountService, TransactionService }
}

package "Presentation" {
  package "UI" { LoginPage, Dashboard, TransactionView }
}

package "Application" {
  package "Controllers" { LoginController, TransactionController }
}

Domain Model --> Presentation   // <> (dependency direction reversed in some notations)
Presentation ..> Domain Model    // dependency: UI depends on domain
Application --> Domain Model
Principle Benefit: Change UI (web → mobile) without touching domain logic.

4.3 Class Diagram

Class diagrams show static structure: classes, attributes, operations, relationships.

4.3.1 Generalization

Inheritance: Child inherits from parent (is-a). Arrow with hollow triangle to parent.

4.3.2 Dependency

Weak usage: One class uses another temporarily (dashed arrow). E.g., parameter, local variable.

4.3.3 Constraints

{OCL or natural language} in curly braces. E.g., {age >= 18} on attribute, or {balance >= 0} on Account.

4.3.4 Composition over Aggregation

Aggregation (hollow diamond): Part independent (e.g., University – Department).
Composition (filled diamond): Part lifecycle tied to whole (e.g., Car – Engine; destroy car → destroy engine).

4.3.5 Singleton Class

Ensures one instance. Private constructor, static getInstance().


class Singleton {
  - static instance: Singleton
  - private Singleton() { }
  + static getInstance(): Singleton {
    if (instance == null) instance = new Singleton();
    return instance;
  }
}

4.3.6 Template Class

Generic class (parameterized). E.g., List, Stack.


// Template (generic) class
class Stack<T> {
  - items: T[]
  + push(item: T)
  + pop(): T
}

abstract class User {

class Customer --|> User   // Generalization

class Order {
