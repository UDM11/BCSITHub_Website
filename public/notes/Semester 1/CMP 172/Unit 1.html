Unit 1: Problem Solving with Computer

1.1 Problem Analysis

Problem analysis is the initial step in solving a problem using a computer. It involves understanding the problem thoroughly and breaking it down into manageable parts. This process ensures that the solution is well-planned and efficient.

  • Identify the Problem: Clearly define what needs to be solved, including inputs, outputs, and constraints.
  • Analyze Requirements: Determine the functional and non-functional requirements of the solution.
  • Break Down the Problem: Divide the problem into smaller sub-problems to simplify the solution process.
  • Evaluate Feasibility: Assess whether the problem can be solved with available resources and technology.

1.2 Algorithms and Flowchart

An algorithm is a step-by-step procedure for solving a problem, while a flowchart is a graphical representation of that algorithm. Both are essential tools for planning and communicating solutions.

  • Algorithm: A finite sequence of well-defined instructions to accomplish a task. For example, to find the sum of two numbers: 
    Input: Two numbers, a and b
Step 1: Read a and b
Step 2: Calculate sum = a + b
Step 3: Output sum
  • Flowchart: Uses symbols like ovals (start/end), rectangles (process), and diamonds (decision) to depict the algorithm visually. For the above algorithm, the flowchart would include:
    • An oval for "Start."
    • A rectangle for "Read a, b."
    • A rectangle for "Sum = a + b."
    • A rectangle for "Output sum."
    • An oval for "End."

1.3 History of C

The C programming language, developed in the early 1970s, is a foundational language in computer science, known for its efficiency and flexibility.

  • Origin: Developed by Dennis Ritchie at Bell Labs between 1972 and 1973 for the UNIX operating system.
  • Evolution: Influenced by earlier languages like BCPL and B. Standardized as ANSI C in 1989 (C89) and later as C99, C11, and C17.
  • Impact: Widely used in system programming, embedded systems, and as a base for languages like C++ and Java.

1.4 Structure of C Program

A C program follows a specific structure to ensure clarity and functionality. A basic C program includes several key components.

#include <stdio.h>
#define PI 3.14

int main() {
    printf("Hello, World!\n");
    return 0;
}
  • Preprocessor Directives: Lines starting with # (e.g., #include, #define) that instruct the compiler to include files or define constants.
  • Main Function: The entry point of the program, where execution begins.
  • Statements: Instructions within curly braces {} that perform the program's tasks.
  • Return Statement: Indicates the program's termination and returns a value to the operating system.

1.4.1 Debugging

Debugging is the process of identifying and fixing errors in a program. Common types of errors include:

  • Syntax Errors: Mistakes in code structure, like missing semicolons. 
    // Syntax error: missing semicolon
printf("Error")
  • Logical Errors: Errors in the program's logic, producing incorrect results. 
    // Logical error: incorrect formula
int area = length + width; // Should be length * width
  • Runtime Errors: Errors during execution, like division by zero.
  • Tools: Debuggers (e.g., GDB), print statements, and IDEs help locate and fix errors.

1.4.2 Compiling a C Program

Compiling converts C source code into machine code that the computer can execute.

  • Steps:
    1. Preprocessing: Expands directives like #include and #define.
    2. Compilation: Converts code to assembly language.
    3. Assembly: Translates assembly code to object code.
    4. Linking: Combines object code with libraries to create an executable.
  • Command: Using a compiler like GCC: 
    gcc -o program program.c

1.4.3 Executing a C Program

Execution runs the compiled program to perform its tasks.

  • Running the Program: On Unix/Linux, use: 
    ./program
    On Windows, run the .exe file directly.
  • Input/Output: Programs may accept user input (e.g., via scanf) and display output (e.g., via printf).
  • Environment: Ensure the system has the necessary runtime environment (e.g., libraries).

1.5 Testing and Documentation

Testing ensures the program works correctly, while documentation explains its functionality for users and developers.

  • Testing:
    • Unit Testing: Tests individual functions or modules.
    • Integration Testing: Tests combined modules.
    • System Testing: Tests the entire program.
    • Test Cases: Include normal, boundary, and error conditions.
  • Documentation:
    • Internal: Comments within code to explain logic. 
      // Calculate area of rectangle
int area = length * width;
    • External: User manuals, design documents, or README files.
    • Importance: Facilitates maintenance and collaboration.

1.6 Preprocessor and Macros

The preprocessor modifies C code before compilation, and macros are shorthand for code snippets.

  • Preprocessor Directives:
    • #include: Imports header files (e.g., #include <stdio.h>).
    • #define: Defines constants or macros. 
      #define PI 3.14
#define SQUARE(x) ((x) * (x))
    • #ifdef, #endif: Conditional compilation based on defined macros.
  • Macros:
    • Object-like Macros: Replace identifiers with values (e.g., #define PI 3.14).
    • Function-like Macros: Replace calls with expressions (e.g., #define SQUARE(x) ((x) * (x))).
    • Caution: Macros lack type checking and can cause side effects if not carefully designed.
